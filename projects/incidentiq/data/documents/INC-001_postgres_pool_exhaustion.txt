Incident INC-001 [P1 - Database]: PostgreSQL Primary Node Connection Pool Exhaustion. Description: At 14:22 UTC the primary PostgreSQL 14 cluster (db-prod-1) began rejecting new connections with 'FATAL: sorry, too many clients already'. PgBouncer pool fill rate hit 100% and cl_waiting climbed past 800. Checkout, search, and account APIs returned 503 for ~18% of requests for 22 minutes; revenue impact estimated at $42K. Detected by Datadog monitor 'pg.connections.utilization > 90%' which paged the DBOC on-call.. Triage Steps: 1) Run: psql -h db-prod-1 -U postgres -c "SELECT count(*), state FROM pg_stat_activity GROUP BY state;" 2) Connect to PgBouncer admin and run: SHOW POOLS; SHOW CLIENTS; SHOW SERVERS; 3) Identify long-running idle-in-transaction sessions: SELECT pid, usename, application_name, state, NOW()-xact_start AS xact_age, query FROM pg_stat_activity WHERE state='idle in transaction' ORDER BY xact_age DESC LIMIT 20; 4) Check max_connections vs current usage: SHOW max_connections; SELECT count(*) FROM pg_stat_activity; 5) Inspect PgBouncer config: cat /etc/pgbouncer/pgbouncer.ini | grep -E 'pool_mode|default_pool_size|max_client_conn' 6) Tail PgBouncer logs for 'closing because' and 'no more connections': journalctl -u pgbouncer -n 500 --no-pager 7) Correlate with application deploys via: kubectl rollout history deployment/analytics-api -n analytics 8) Capture pg_locks snapshot: SELECT locktype, mode, count(*) FROM pg_locks GROUP BY locktype, mode ORDER BY count DESC;. Root Cause: A scheduled analytics ETL deployed in release v3.8.0 opened explicit transactions but never committed when its downstream Kafka producer raised an exception. Connections were returned to PgBouncer as 'idle in transaction', exhausting the 200-slot transaction pool within 14 minutes of the ETL trigger.. Resolution: 1) Terminate offending idle-in-transaction sessions: SELECT pg_terminate_backend(pid) FROM pg_stat_activity WHERE state='idle in transaction' AND NOW()-xact_start > interval '5 minutes'; 2) Temporarily raise PgBouncer default_pool_size from 200 to 350 and reload: pgbouncer -R /etc/pgbouncer/pgbouncer.ini 3) Roll back analytics-api to v3.7.4: kubectl rollout undo deployment/analytics-api -n analytics --to-revision=42 4) Set statement_timeout=30s and idle_in_transaction_session_timeout=120s on the analytics DB role: ALTER ROLE analytics_etl SET idle_in_transaction_session_timeout = '120s'; 5) Verify pool utilisation drops below 60% in Datadog dashboard 'PostgreSQL — Connections' 6) Deploy v3.8.1 with try/finally guard ensuring conn.rollback() and conn.close() in producer exception path 7) Restore default_pool_size to 200 once new build is stable for 30 minutes 8) Add Datadog monitor 'pg.idle_in_transaction.duration > 60s' paging DBOC at warning threshold. MTTR: 28 minutes. Lessons Learned: Application code must guarantee transaction termination on every exception path; connection pools are a hard ceiling, not a soft target. Add idle_in_transaction timeouts at the role level so a single buggy service cannot consume the cluster..
